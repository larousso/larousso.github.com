# Property based testing en java

Le "property based testing" est un type de tests souvent utilisé en programmation fonctionnelle pour valider des lois.
Il y a quelques années, j'avais testé `jqwik`, une librairie de property based testing en java, pour valider des lois sur une lib d'optics en java (ouais je sais, ça à l'air de ne rien vouloir dire cette phrase). Je n'étais pas allé plus loins que ça.

## C'est quoi, le property based testing ?

En programmation fonctionnelle, certains concepts sont validés par des lois.
On va prendre l'exemple d'une `Monoid`, un mot barbare ,pour, parler en gros de concaténation. Une `Monoid` peut se définir ainsi :

```java
interface Monoid<T> {
    T empty();
    T combine(T a, T b);
}
```

On peut ensuite créer des instances :

```java
Monoid<String> stringMonoid = new Monoid<String>() {
    String empty() {
        return "";
    }
    String combine(String a, String b) {
        return a + b;
    }
};
```

Ou encore

```java
<T> Monoid<List<T>> listMonoid() {
    return new Monoid<List<T>>() {
        List<T> empty() {
            return List.of();
        }
        List<T> combine(List<T> a, List<T> b) {
            return a.appencAll(b);
        }
    };
}
```

Mais ça n'est pas suffisant ! Il y a des lois à respecter !

Pour que mon instance soit vraiment une `Monoid`, il faut que :
* `combine(a, empty()) == combine(empty(), a) == a` : vérifier l'élément neutre
* `combine(combine(a, b), c) == combine(a, combine(b, c))` : avec ça on peut paralléliser (si si je vous jure)

Le problème, c'est que jouer un seul test pour vérifier ça n'est pas très probant, c'est ici, qu'entre en scène, le property based testing !

Avec jqwik, on peut écrire ça :

```java
void neutralLaw(@ForAll String aString) {
    assertThat(stringMonoid.combine(aString, empty()))
            .isEqualTo(stringMonoid.combine(empty(), aString))
            .isEqualTo(aString);
}
void associativityLaw(@ForAll String a, @ForAll String b, @ForAll String c) {
    assertThat(stringMonoid.combine(a, stringMonoid.combine(b, c)))
            .isEqualTo(stringMonoid.combine(stringMonoid.combine(a, b), c));
}
```

Jqwik va alors, exécuter ces tests avec des données random un certain nombre de fois. On va alors valider que statistiquement la loi est juste.

## Tester du code métier

Pour tester du code métier il va falloir trouver comment gérer les assertions. On va devoir trouver une sorte de loi pour valider notre métier.

Per exemple, si je veux tester la création d'une personne dans mon API, ce que je veux, c'est que :
* la réponse ait un status http ok
* si je relis la personne, on doit avoir les mêmes données que ce qui a été envoyé

Ainsi, on va valider que la création fonctionne quelles que soient les données envoyées (dans le bon format)
et, en relisant la donnée, on va vérifier, qu'il n'y a pas de perte en route.


### La mise en œuvre

Pour faire nos tests, on va utiliser [jqwik](https://jqwik.net/). Jqwik a son propre runner,
il est parfait pour des tests unitaires, mais moins adéquat pour les tests d'intégration avec spring.

A priori il existe un module spring mais il n'a pas l'air de fonctionner avec le mode `@SpringBootTest`. On va donc mixer `jqwik` et `jupiter`.

Dans Jupiter, il existe les `@ParameterizedTest` qui permet d'exécuter un jeu de test paramétré. On peut utiliser ce concept pour injecter des données générées à part de `jqwik`.

### Générer des cas de test

Dans notre cas, on veut générer des personnes, qui ont le format suivant :

```java
@Builder
record Person(@NotNull UUID id, Option<String> lastName, Option<String> firstName, Option<Gender> gender, List<Contact> contacts) {
    public Person {
        lastName = Objects.requireNonNullElse(lastName, Option.none()).map(n -> StringUtils.upperCase(n));
        firstName = Objects.requireNonNullElse(firstName, Option.none()).map(n -> WordUtils.capitalizeFully(n, "-", " "));
        gender = Objects.requireNonNullElse(gender, Option.none());
        contacts = Objects.requireNonNullElse(contacts, List.empty());
    }
}
```

```java
enum Gender {
    MALE, FEMALE, ...;
}
```

```java
sealed interface Contact {
    record EmailContact(@NotNull @Email String email) implements Contact {}
    record PhoneNumberContact(@NotNull @PhoneNumber String phoneNumber) implements Contact {}
    record AddressContact(@NotNull String streetAddress, @NotNull @ZipCode String zipCode, @NotNull String city) implements Contact {}
}
```

Aller, c'est parti, on va commencer à générer de la donnée ! `Jqwik` a un certain nombre d'utilitaires pour générer des valeurs de tout type.

Pour le nom et le prénom, on va envoyer des lettre majuscules, minuscule, le tiret et l'espace
```java
static Arbitrary<String> lastName() {
   return Arbitraries.strings()
                .withCharRange('a', 'z')
                .withCharRange('A', 'Z')
                .withChars('-', ' ')
                //On limite la taille
                .ofMinLength(1)
                .ofMaxLength(100)
                //On injecte 50% de valeurs nulles
                .injectNull(0.5);
}
```

Pour le genre, c'est facile, c'est une liste finie :
```java
static Arbitrary<Gender> gender() {
    return Arbitraries.of(Gender.values());
}
```

Pour les contacts, ça être un peu plus compliqué !

On commence par l'email, il existe un module pour ça :
```java
static Arbitrary<Contact> emailContact() {
    return Web.emails().map(EmailContact::new);
}
```

Pour le téléphone, on peut choisir une autre approche afin d'avoir des téléphones plus proche de la réalité : [datafaker](https://www.datafaker.net/)
```java
static Faker faker = new Faker(Locale.of("fr"));

static Arbitrary<Contact> phoneNumberContact() {
    return Arbitraries
        .create(() -> faker.phoneNumber().phoneNumber())
        .map(PhoneNumberContact::new);
}
```

Pour l'adresse, on a aussi besoin de données cohérente, on va utiliser [datafaker](https://www.datafaker.net/)

```java
static Arbitrary<Contact> addressContact() {
    return Arbitraries.create(() -> {
        Address address = faker.address();
        return AddressContact.builder()
                .city(address.city())
                .streetAddress(Option.of(address.streetAddress()))
                .zipCode(address.zipCode())
                .build();
    });
}
```

Un contact est un type union, c'est soit une adresse, soit un email, soit un téléphone. En plus de ça, il faut générer une liste, voici la méthode :

```java
static Arbitrary<List<Contact>> contacts() {
    return Arbitraries
        .oneOf(
            emailContact(),
            phoneNumberContact(),
            addressContact()
        )
        .list();
}
```

Tout est presque prêt, on va maintenant créer une personne :

```java
static Arbitrary<Person.PersonBuilder> personProperties() {
    return Combinators.combine(
            lastName(),
            firstName(),
            gender(),
            contacts()
    ).as((lastName, firstName, gender, contacts) ->
            Person.builder()
                .lastName(Option.of(lastName))
                .firstName(Option.of(firstName))
                .gender(Option.of(gender))
                .contacts(contacts)
    );
}
```

Il reste à écrire le test lui-même :

```java
@ParameterizedTest
@MethodSource("providePerson")
public void createPersons(Person.PersonBuilder personBuilder) {
    WebClient webClient = this.webClientBuilder.build();

    UUID personId = UUID.randomUUID();
    var personToCreate = personBuilder.id(personId.toString()).build();

    var created = webClient.post()
                .uri(url("/api/persons"))
                .bodyValue(Json.toJson(personToCreate, PersonJsonFormat.personWrite()))
                .retrieve()
                .toEntity(Person.class)
                .map(r -> r.getBody())
                .block();
    var get = webClient.get()
                .uri(url("/api/persons"))
                .retrieve()
                .toEntity(Person.class)
                .map(r -> r.getBody())
                .block();
    assertThat(personToCreate).isEqualTo(created);
    assertThat(personToCreate).isEqualTo(get);
}
```
